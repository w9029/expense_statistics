# iOS 端开发计划

## 目标

基于现有 `project/iosApp` React Native 工程，开发一版遵循 iOS 使用习惯、功能覆盖前端现有能力的原生感移动 App。

本计划参考了：

- 前端页面与路由
- `packages/api-client` 与 `packages/domain`
- 后端 `/api/v1` 路由
- 当前 `iosApp/docs/commit-log.md`

## 范围

本期目标覆盖当前前端已经完成的核心功能：

- 登录
- 注册
- 邀请接受
- 账本列表
- 账本详情与消费列表
- 普通消费创建/编辑
- 合并消费创建/编辑
- 分类管理
- 协作与邀请管理
- 数据分析
- 个人资料与默认账本
- 多语言

说明：

- Web 的落地宣传页 `landing-page` 不作为 iOS 首期重点页面照搬。
- iOS 端会按移动端习惯重组导航与交互，不会 1:1 复刻网页布局。

## 总体原则

1. 先打通基础架构，再做业务页面。
2. 先做高频主链路，再做管理型页面。
3. 先做普通消费，再做合并消费。
4. 所有用户可见文本从第一天开始接入多语言。
5. 尽量复用 `packages/api-client` 与 `packages/domain`，避免 iOS 端重复定义接口模型。

## 推荐开发顺序

### 0. 基础架构

对应内容：

- React Native 工程结构整理
- 应用导航
- 认证状态管理
- API Client 接入
- 多语言
- 全局错误/加载/Toast

需要完成：

- 建立 `src` 目录与基础分层，例如 `screens`、`features`、`services`、`navigation`、`i18n`
- 接入共享的 `api-client` / `domain`
- 统一会话存储、刷新 token、401 处理
- 建立 iOS 风格导航骨架：公开流、认证流、主应用流
- 建立中英文/日文资源与语言切换能力

### 1. 认证与邀请入口

参考前端页面：

- `login-page`
- `register-page`
- `invitation-page`

需要完成：

- 登录
- 三步注册流程：发验证码、验码、补资料注册
- 登录后跳转逻辑
- 已登录用户接受邀请
- 未登录用户从邀请页进入登录/注册后回跳

这是第一条完整用户链路，完成后 App 才真正可用。

### 2. 账本列表

参考前端页面：

- `account-books-page`

需要完成：

- 账本列表展示
- 创建账本
- 设为默认账本
- 删除账本
- 离开账本
- 从账本列表进入具体账本

这里建议直接做成 iOS 首页，而不是保留 Web 的中转感布局。

### 3. 个人资料

参考前端页面：

- `profile-page`

需要完成：

- 修改用户名、偏好货币、语言、头像路径
- 修改默认账本
- 会话中的用户信息同步更新

这个模块依赖较少，适合在账本列表之后尽快完成。

### 4. 账本详情与消费列表

参考前端页面：

- `account-book-detail-page`

需要完成：

- 账本基本信息展示
- 消费列表展示
- 分页
- 关键筛选：关键词、成员、币种、金额区间、日期区间、分类
- 账本元信息编辑
- 删除账本 / 离开账本入口
- 普通消费 / 合并消费入口

这是主工作台，也是后续多个模块的依赖中心。

### 5. 分类管理

参考前端页面：

- `expense-categories-page`

需要完成：

- 分类列表
- 普通分类 / 合并分类分组展示
- 新建分类
- 编辑分类
- 删除分类
- 颜色与类型展示

这一模块完成后，消费录入的体验才会稳定。

### 6. 普通消费

参考前端页面：

- `normal-expense-page`

需要完成：

- 新建普通消费
- 编辑普通消费
- 表单校验
- 创建后返回账本
- 创建并继续下一条

普通消费是最常用链路，优先级高于合并消费。

### 7. 合并消费

参考前端页面：

- `merged-expense-page`

需要完成：

- 新建合并消费
- 编辑合并消费
- 父项与子项表单
- `pretax` / `posttax` 两种录入模式
- 子项动态增删
- 金额预览与差额提示
- 创建并继续下一条

这一块表单复杂度最高，建议放在普通消费之后单独攻坚。

### 8. 协作与邀请管理

参考前端页面：

- `account-book-collaboration-page`

需要完成：

- 成员列表
- 创建邀请
- 复制邀请链接
- 删除邀请
- 转移所有者
- 移除成员
- 非 owner 用户离开账本

这个模块偏管理操作，优先级低于日常记账主链路，但首期应完成。

### 9. 数据分析

参考前端页面：

- `analytics-page`
- `features/analytics/category-share-card`

需要完成：

- 分类占比
- 消费趋势
- 日 / 月维度切换
- 日期范围切换
- 分类筛选

建议在账本详情稳定后接入，因为它复用账本、分类和金额展示能力。

### 10. 收尾与发布准备

需要完成：

- 空态、错误态、加载态统一整理
- 文案与多语言补齐
- 真机网络与异常场景检查
- 列表与表单交互优化
- HTTP 场景下的 ATS 配置复核
- App 图标、启动图、基础元信息整理

## 建议的实际提交粒度

为了和现有 `commit log` 风格保持一致，后续建议按下面的粒度推进：

1. 基础架构与目录整理
2. 认证与邀请入口
3. 账本列表与个人资料
4. 账本详情与消费列表
5. 分类管理
6. 普通消费
7. 合并消费
8. 协作与邀请管理
9. 数据分析
10. 收尾优化

## 当前结论

如果按“先能用、再完整、再复杂”的顺序推进，最合理的 iOS 开发主线是：

`基础架构 -> 认证/注册/邀请 -> 账本列表 -> 个人资料 -> 账本详情 -> 分类 -> 普通消费 -> 合并消费 -> 协作 -> 分析 -> 收尾`

这个顺序基本贴合前端页面能力，同时也符合 iOS 端实际使用路径。
